03

Key Learnings

What we've discovered along the way

01

Hyperbase Isn't AI-Ready Yet

The biggest friction isn't PM skill—it's our codebase. "It feels like there is no ceiling—but we can't bring this into Hyperbase because of these gaps."

1
No Claude MD / skills files. No indexing or directory structure for AI tools. Context fills up too fast.
2
No documentation ownership. Code is diffuse across 50+ locations with no coherent team ownership.
3
Overloaded terms. Words like "future" and "permission" mean different things, causing AI to go down wrong paths.

These gaps affect everyone—not just PMs. Fixing them would unlock AI productivity across the entire engineering org.

These are solvable. Product urges continued focus on codebase skills, indexes, and teams owning .md files for their areas.

02

Prototyping Changes the Game

The workflow shift from "write PRD, go through design" to "prototype first, then refine" produces better starting points and faster iteration.

"

This was way better than writing a PRD and going through design. Now the starting point with which I go to design is much stronger.

— PM after the prototyping spike

The Governance spike proved this works even in our most complex areas—permissions and licensing, arguably our most complicated domain. If it works there, it works elsewhere.

Instead of describing what you want, show working code. Engineers receive a functional starting point instead of static mockups.

03

PM-Eng Pairing Creates New Value

PMs and engineers approach AI tools differently. These differences are complementary, not competing.

"

It was incredibly helpful to see how you approach using Claude Code as it is a departure from how I solve building as an engineer.

— Engineer after pairing with a PM

PMs describe outcomes; engineers specify implementations. When paired, they can validate approaches faster and catch blind spots earlier. This is a new collaboration model worth exploring.

Consider dedicated PM-Eng pairing sessions for AI-assisted development. Both roles learn from each other's mental models.

04

Our Review Process Needs to Adapt

PMs can write code faster than engineers can review it. Our current PR process assumes authors can iterate independently on feedback.

PM Writes Code Fast
ENG Reviews Blocked

Options to explore: Bugbot for automated pre-review, dedicated review slots for PM code, or pairing during the review itself to accelerate iteration.

The 80% solution shipped in hours beats the 100% solution waiting months in backlog. Process should enable this.

04

Where We Are

What we've learned and what we still need to answer

What We Know

PMs Can Ship Production Code

The hypothesis is validated. PMs with AI tools can write and ship production-ready code—even in complex areas like governance and permissions.

  • Prototyping accelerates the design/eng workflow
  • PM-Eng pairing creates complementary value
  • The 80% solution shipped fast beats 100% in backlog

But Hyperbase isn't AI-ready yet, and our review process needs to adapt for this to scale.

Status Validated
Open Questions

Should PMs Do This? How Much?

We've proven PMs can ship code. The harder question is whether they should—and how much of their time this deserves.

What's the right balance between coding and traditional PM work?
  • When is PM coding ROI-positive vs. opportunity cost?
  • What types of work are best suited for PM coding?
  • How do we measure value created vs. time invested?
Status Needs Answer
