03

Key Learnings

What we've discovered along the way

01

PMs May Actually Be Better at AI Coding

Engineers' deep expertise can actually work against them with AI tools. PMs approach problems differently—describing what they want rather than how to build it—which aligns better with how AI coding assistants work.

"

As a career engineer I cannot think that way yet.

— Engineer after watching a PM complete Field Agent control in one day

The PM mindset of "describe the outcome" rather than "specify the implementation" may be a feature, not a bug, when working with AI.

02

Negative Instructions Beat Proactive Ones

When configuring AI coding assistants (via Claude.md files), telling it what NOT to do is more effective than describing ideal behavior.

Less Effective "Always ask before pushing code"
More Effective "NEVER push without explicit user approval"

The AI responds more reliably to clear prohibitions than aspirational guidance. This mirrors how we train humans on safety-critical procedures.

03

Code Review is the Real Bottleneck

PMs can write code faster than engineers can review it. The traditional code review process assumes the author is an engineer who can iterate independently.

PM Writes Code Fast
ENG Reviews Blocked

We need different review workflows for PM-authored code—possibly automated pre-review tools like Bugbot or dedicated review slots.

04

Scoping Down is a Legitimate Strategy

The instinct to "just fix it properly" often leads to scope creep that exceeds PM coding capabilities.

PM can ship (80% value)
Eng needed

Explicitly choosing a smaller scope—even if it means shipping a partial solution—is valid when it delivers user value faster. Sometimes a PM can ship the 80% solution in hours while the 100% solution waits months.

05

Keep a Stable Plan Doc Outside Claude

Claude sessions compact, crash, or need to restart. Without an external plan document, you lose context and have to re-explain everything.

Pain Point "Had to start/end many sessions and kept losing context"
Solution Maintain a plan.md file that survives session restarts

Your plan doc becomes the source of truth. When Claude compacts or you need a fresh session, you can paste the plan back in and pick up where you left off.

04

Early Conclusions

What we believe to be true based on our findings

Conclusion #1

PMs Can Produce ROI-Positive Prototypes

The governance framework audit alone demonstrates that PMs with AI tools can deliver meaningful technical output. The work completed would have taken engineering time that wasn't available.

The question isn't "can PMs code?" but "what should PMs code?"

Even if PM code requires more review cycles, the net ROI is positive for certain categories of work—particularly internal tools, audits, and well-scoped bug fixes.

Verdict Validated
Conclusion #2

AI Accelerates Coding But Doesn't Simplify Engineering

Claude Code dramatically reduces the barrier to writing syntactically correct code. However, understanding system architecture, anticipating edge cases, and navigating organizational code patterns remain genuinely difficult.

Writing Code
Easier with AI
Understanding Systems
Still Hard

The multi-select dropdown case study proves this: what looked like a 2-hour fix became 6 hours because the complexity wasn't in the code—it was in understanding how the system behaves.

Verdict Nuanced Reality
