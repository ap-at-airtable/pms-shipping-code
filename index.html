What we've discovered along the way

01

Hyperbase Isn't AI-Ready Yet

The biggest friction isn't PM skill‚Äîit's our codebase. "It feels like there is no ceiling‚Äîbut we can't bring this into Hyperbase because of these gaps."

1
CLAUDE.md hasn't been updated in months. Team members don't feel empowered to add learnings or "do not do X" statements.
2
Skills usage is very limited. No clear path for incorporating learnings into the shared codebase environment.
3
No documentation ownership. Code is diffuse across 50+ locations with no coherent team ownership.
4
Overloaded terms. Words like "future" and "permission" mean different things, causing AI to go down wrong paths.
5
Code is scattered across 50+ locations. Absent or unclear code ownership creates diffusion problems, making domain-specific Claude MD files impractical.

These gaps affect everyone‚Äînot just PMs. Fixing them would unlock AI productivity across the entire engineering org.

Proposed Strategy: Team-Owned Claude MD
  • Focus on "do not do X" statements ‚Äî negative instructions are more effective than proactive ones
  • Break into team-owned indexes ‚Äî roughly one per complex area/pod, append-only for scalability
  • Claude can traverse an index without overloading context until it's in the right "space"
  • Use on-call structure for recurring input to skill/agent/claude files

Bright spot: The UX content team created a helpful content and copy skill. This is the model to follow.

02

Airgapped Prototyping is the Untapped Opportunity

The highest-value application of PM coding might not be production code at all. It's rapid prototyping for user validation.

The Approach (from Joseph Soltzberg)
  • Create a separate repo completely airgapped from production
  • Include styling libraries that mimic the real product
  • Build functional prototypes and distribute for unmoderated testing at scale (50-200 users)
  • Use AI to analyze results and generate learnings

This approach lets teams "climb the truth curve" much more efficiently‚Äîvalidating ideas before committing significant engineering resources.

This was underexplored in the spike. Most effort went into production code. The prototyping path may have higher ROI.

03

Code Quality Requires Active Supervision

Claude makes suboptimal choices for production-quality code. You can't just accept the output‚Äîyou need to actively supervise.

‚ö†Ô∏è Struggles with string handling and makes arbitrary code structure decisions
‚ö†Ô∏è Complex areas (multiple feature flags + admin features) cause confusion
‚ö†Ô∏è DB migrations and edge cases are "scary territory" requiring design help
‚ö†Ô∏è Sometimes digs into features that aren't relevant to the task
‚ö†Ô∏è Tests and CI are a slog ‚Äî Playwright testing proved difficult, PRs failing in CI with unclear Buildkite debugging

The implication: PM coding isn't "hands off." It requires reading and understanding every line before shipping.

04

Design Review is Non-Negotiable

Any user-facing UI needs design review before shipping. This was a blind spot in early thinking about PM coding.

"

PMs shipping code must maintain Airtable's design quality and consistency. This requires structured design collaboration throughout the development process.

‚Äî Kristen Freitas, Design POV
Key Design Findings
Late design review is expensive. At the PR stage, designers must do mini-explorations to understand context, sometimes reverse-engineering the PR to understand the problem through the solution.
Simple changes aren't always simple. Small changes can have ripple effects impacting multiple surfaces and creating inconsistencies with existing patterns.
Two categories of feedback: Design polish (padding, margins, Aero component usage, color, alignment) and UX pattern issues (inconsistencies requiring auditing, exploration, and potentially different approaches).
Proposed Design Checkpoints
1
Planning

Design adds to lightweight PRD before coding. UX pattern guidance, Aero components, a11y/l10n, error states. Quick mocks to feed vibe-code.

2
Prototype

Feedback on working 1st draft. Aero usage, UI consistency, UX flow, adjacent surface impact. Weekly crit for awareness.

3
Crit

Design signoff before dogfood. Async for minor changes, weekly crit for larger features, domain expert when possible.

Design Resources
Kristen Freitas ‚Äî Main point of contact for design review
Trijeet Mukhopadhyay ‚Äî Prototyping expertise, extensive UI prototyping experience
Chloe's workshop ‚Äî Working with design, Storybook, and QA-ing UI

Recommendation: Create skill docs for design (content team's tone/voice skill is the model), invest in getting more Aero components into code.

Our Review Process Needs to Adapt

PMs can write code faster than engineers can review it. Our current PR process assumes authors can iterate independently on feedback.

PM Writes Code Fast

Options to explore: Bugbot for automated pre-review, dedicated review slots for PM code, or pairing during the review itself to accelerate iteration.

The 80% solution shipped in hours beats the 100% solution waiting months in backlog. Process should enable this.

PM Coding Playbook

Guidelines for product managers shipping code

PM coding is a tool for acceleration, not a replacement for engineering. Use it to unblock small things quickly, validate ideas faster, or get a head start. Don't use it to become a shadow engineer.

◆ Before You Start: Three Questions

1

Is this the right type of work?

‚úì Good Candidates
  • Paper cuts and small UX fixes
  • Copy/label changes
  • Small, isolated bug fixes
  • Functional prototypes
  • Error message improvements
‚úó Poor Candidates
  • Multiple feature flags
  • Database migrations
  • Complex edge cases
  • Security-sensitive areas
  • Core infrastructure
2

Can I scope this tightly?

Before starting, write down:

  • What exactly will I change?
  • What files will I touch?
  • What's the acceptance criteria?
  • What's my time budget?

If you can't answer these clearly, it's probably too ambiguous.

3

Is this the best use of my time?

PM coding has opportunity cost. You could be:

  • Talking to customers
  • Working on strategy
  • Unblocking your team
  • Prioritizing the backlog

Only code when it's genuinely higher ROI.

▶ The Two Tracks

Track A

Production Code

Papercuts & Quality-of-Life in Hyperbase

For minimal full-stack work, a PM should feel empowered to make enhancements with eng support. This frees up backlog and core eng focus ‚Äî not about PMs becoming junior devs, but increasing overall velocity.

The Workflow
  1. Scope first ‚Äî Define exactly what you're changing
  2. Set time budget ‚Äî Typically 2-4 hours max
  3. Build it ‚Äî With active AI supervision
  4. Self-review ‚Äî Read and understand every line
  5. Design review ‚Äî Loop in design for UI changes
  6. Submit PR ‚Äî Expect 1-2 review cycles
  7. Hand off if stuck ‚Äî After 2 cycles, reassess
Guardrails
‚è±Ô∏è Time limit: Stop after budget, even if not done
üîÑ Review limit: After 2 cycles, reassess
üö® Complexity trigger: Feature flags, migrations, edge cases ‚Äî stop
üëÄ Supervision: Never blindly accept AI output
Track B

Prototyping

The Physical Form of the PRD

When there's hesitation on priorities or broader buy-in, prototyping can clarify vision and impact. It makes it clearer across design and eng what we're actually building.

The Workflow
  1. Define hypothesis ‚Äî What are you trying to learn?
  2. Build prototype ‚Äî Functional, not production quality
  3. Deploy for testing ‚Äî Password-gated, separate
  4. Get user feedback ‚Äî Unmoderated testing at scale
  5. Synthesize learnings ‚Äî What's the recommendation?
  6. Hand off ‚Äî With learnings and prototype
Guardrails
üéØ Learning-focused: Goal is insight, not shippable code
üóëÔ∏è Disposable: This code won't ship ‚Äî that's fine
üìä Measure: Track what you learned, not what you built

★ Claude Code Best Practices

üìã

Plan Mode

Use /plan before any non-trivial change. Claude will explore the codebase, identify relevant files, and propose an approach before writing code.

When to use:
  • Changes touching multiple files
  • Unfamiliar areas of codebase
  • When you're not sure where to start
üîç

Code Review

Use /review to have Claude review your changes before submitting a PR. Catches issues early and improves code quality.

What it checks:
  • Style guide violations
  • Potential bugs and edge cases
  • Missing error handling
üíæ

Commit Often

Use /commit to checkpoint your work. Small, frequent commits let you rollback if Claude goes down a wrong path.

Pro tip:
  • Commit after each working state
  • Use descriptive messages
  • Makes it easy to revert mistakes
üìÅ

CLAUDE.md Files

These files give Claude codebase context. Read them to understand patterns. Add learnings when you discover something useful.

You can contribute:
  • "Do not do X" warnings
  • Pattern examples
  • Common gotchas

Effective Prompting

‚úì Good Prompts
"Fix the padding on the submit button in LoginForm.tsx" Specific file, specific element, specific issue
"Add a loading spinner while fetchUserData is pending" Clear outcome, references existing code
"The error message says X but should say Y" Concrete before/after
‚úó Avoid These
"Make the UI better" Too vague ‚Äî better how?
"Fix the bug" Which bug? What's the symptom?
"Refactor this to be cleaner" Subjective ‚Äî leads to arbitrary changes

Common Pitfalls in Hyperbase

‚ö†Ô∏è
Overloaded terms

Words like "permission," "future," and "field" mean different things in different contexts. Be explicit about which one you mean.

‚ö†Ô∏è
Context limits

Complex areas fill up Claude's context fast. If responses get confused or repetitive, start a new conversation.

‚ö†Ô∏è
Feature flag intersections

Areas with multiple feature flags are hard for Claude (and humans). Consider these off-limits for PM coding.

‚ö†Ô∏è
String handling

Claude often makes suboptimal choices with string manipulation. Review these carefully.

➜ When to Hand Off

Signal Action
3+ review cycles Hand off ‚Äî past the efficiency point
Scope expanded Hand off ‚Äî bigger than estimated
Hit DB migration Hand off ‚Äî scary territory
Feature flag complexity Hand off ‚Äî AI and PMs both struggle
"I don't know why this works" Hand off ‚Äî you need to understand what ships

How to Hand Off Well

  • Document what you tried and learned
  • Leave the PR/branch in a clean state
  • Explain where you got stuck and why
  • Offer to pair to transfer context

✓ Quick Reference Checklist

Before Starting

  • Is this a good candidate? (small, scoped, isolated)
  • Can I clearly define scope and acceptance criteria?
  • Have I set a time budget?
  • Is this higher ROI than other PM work?

While Working

  • Am I actively supervising AI output?
  • Am I within my time budget?
  • Have I hit any complexity triggers?

Before Submitting PR

  • Have I read and understood every line?
  • Have I gotten design review on UI changes?
  • Is the scope still what I defined?

After Review

  • Have I stayed within 2 review cycles?
  • If stuck, have I handed off cleanly?
  • Have I documented what I learned?

The Meta Point

This playbook exists because PM coding is situational, not universal.

The value comes from:
  • Unblocking small things that would otherwise wait
  • Learning faster through functional prototypes
  • Building empathy with the codebase
The danger is:
  • Spending PM time on things eng should do
  • Shipping lower-quality code
  • Neglecting core PM responsibilities

Use judgment. When in doubt, don't code ‚Äî PM.
